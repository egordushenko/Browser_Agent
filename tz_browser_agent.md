# ТЗ: Автономный AI-агент для автоматизации браузера

> Документ-задание для исполнителя (Codex/Claude Code). Пиши код инкрементально по milestones (раздел 13), коммить после каждого. Все секреты — через `.env`, не хардкодить. Ключевые анти-требования (раздел 12) — жёсткие: их нарушение = провал задания.

---

## 1. Цель и что реально проверяется

Собрать **автономного агента**, который по текстовой задаче на естественном языке сам управляет **видимым** браузером и решает многошаговую задачу с переходами между страницами, останавливаясь только когда нужна информация от пользователя, требуется подтверждение необратимого действия, или задача выполнена.

Проверяется не знание библиотек, а: перцепция страницы без хардкода, агент-луп на tool calling, управление контекстом в рамках токен-бюджета, и минимум один продвинутый паттерн (у нас — sub-agent + error-handling + security).

**Главное, что грейдят (по убыванию веса, прикидка):**
1. Перцепция и выбор элементов **в рантайме** (селекторы не захардкожены).
2. Управление контекстом (в модель не летит целая страница).
3. Автономный луп (агент сам решает следующий шаг, а не идёт по заготовке).
4. Продвинутые паттерны: sub-agent, обработка ошибок, security-гейт.

---

## 2. Форма конечного продукта

Локальное CLI-приложение. При запуске одной командой:
- Поднимается **видимый (headed, не headless)** Chromium через Playwright с persistent-профилем.
- Открывается терминал-REPL — туда пользователь пишет задачу и там же видит live-лог: какие инструменты агент вызывает и с какими аргументами, что отвечает sub-agent, к какому решению приходит.
- Пользователь наблюдает действия одновременно в окне браузера и в терминале.
- В конце агент выдаёт краткий отчёт: что сделано.

Референс-решение (получившее оффер) выглядит именно так: браузер + терминал рядом, в терминале — поток `Using tool: ... Input: {...} Result: ...`, между ними — работа DOM sub-agent, финал — сводка с явной остановкой перед оплатой. Наша цель — та же **форма и архитектурный паттерн**, но собственная реализация (не клон имён тулов и структуры).

## 3. Технологический стек

| Слой | Выбор | Обоснование |
|---|---|---|
| Язык | TypeScript / Node.js (LTS) | Playwright нативно на TS, плотнее экосистема |
| Браузер | Playwright, `launchPersistentContext`, `channel: 'chrome'`, headed | persistent-сессии + меньше антибот-детекта, чем у чистого Chromium |
| LLM (оркестратор) | OpenAI GPT-5.4 Mini | дешёвый пол, хватает на планирование + tool use |
| LLM (DOM sub-agent) | OpenAI GPT-5.4 Nano | узкая задача извлечения → самая дешёвая модель |
| Провайдер | абстракция над LLM (см. 4.6) | ТЗ требует Claude ИЛИ OpenAI; абстракция даёт свободу переключения |

**Цены (проверено ~июль 2026, per 1M токенов вход/выход):** OpenAI GPT-5.4 Mini $0.75/$4.50, Nano $0.20/$1.25, GPT-5.4 $2.50/$15. Альтернатива на Anthropic: Haiku 4.5 $1/$5. Провайдер и модель задаются в `config.ts`/`.env`, чтобы менять без правки кода. **Перед выбором проверить, принимает ли выбранный tier изображения на вход** — нужно, только если решим кормить скриншоты в модель (см. 4.4).

**Прикидка стоимости одного прогона демо-задачи** (допущения, не факт): ~15–25 шагов, при обрезке истории и текстовой перцепции — порядка $0.05–0.30 на прогон. Реальную цифру мерить по факту через usage API.

## 4. Архитектура

```
Пользователь (терминал)
        │  задача на NL
        ▼
┌─────────────────────────────────────────────┐
│ Orchestrator (главный агент-луп)             │
│  observe → LLM выбирает tool → execute →     │
│  результат → повтор, до done/ask_user/limit  │
└───────┬───────────────────────┬─────────────┘
        │ вызывает тулы          │ query_dom(NL-вопрос)
        ▼                        ▼
┌───────────────┐        ┌──────────────────────┐
│ Browser layer │        │ DOM Sub-agent         │
│ (Playwright)  │◄───────│ читает страницу,      │
│ persistent    │  DOM   │ отдаёт текст+селектор │
└───────────────┘        └──────────────────────┘
```

### 4.1. Orchestrator (главный луп)

ReAct-подобный цикл. На каждой итерации:
1. Собрать **компактное наблюдение**: URL, `document.title`, краткое состояние + результат предыдущего тула. **Сырой HTML/DOM сюда не попадает.**
2. Отправить в LLM: системный промпт (стабильный, кэшируется) + tool schemas + обрезанную историю + текущее наблюдение.
3. Получить один tool call, выполнить, записать результат в историю.
4. Повтор до условия остановки.

**Условия остановки:** тул `done` (агент считает задачу выполненной), тул `ask_user` (нужны данные), срабатывание security-гейта (пауза на подтверждение), либо защитные лимиты (раздел 8).

Никакого предзаписанного плана под конкретную задачу. Агент решает шаг в моменте на основе наблюдения.

### 4.2. Набор инструментов (generic-примитивы)

Только атомарные действия. Никаких `apply_to_vacancy`/`checkout`-заготовок.

| Тул | Аргументы | Действие |
|---|---|---|
| `navigate` | `url` | переход по URL / back / forward |
| `query_dom` | `question` (NL) | делегирует DOM sub-agent; возвращает текст: есть ли элемент, его селектор, контент |
| `click` | `selector` | клик по элементу (селектор получен от sub-agent) |
| `type` | `selector`, `text` | ввод текста в поле |
| `scroll` | `direction`, `amount?` | прокрутка |
| `read_page` | `question?` | извлечь релевантный текст (может быть частным случаем query_dom) |
| `wait` | `seconds` \| `condition` | ожидание загрузки/элемента |
| `screenshot` | `full_page?` | скрин (для видимого лога + опц. vision) |
| `ask_user` | `question` | пауза, вопрос пользователю в терминал |
| `done` | `summary` | финал с отчётом |

`click`/`type`/`navigate` при семантике необратимого действия проходят через security-гейт (раздел 9) **до** выполнения.

### 4.3. DOM Sub-agent

Отдельный модуль. **Именно здесь заперт сырой DOM — он не выходит в контекст оркестратора.**

- **Вход:** NL-вопрос оркестратора (напр. «есть ли поле поиска и какой у него селектор?») + компактное представление страницы.
- **Выход:** текстовый ответ + при необходимости **конкретный селектор**, найденный в рантайме.
- **Модель:** дешёвая (Nano).
- **Приоритет селекторов:** стабильные стратегии сначала — `id`, `data-testid`/`data-*`, `role + accessible name`, `text`; хрупкие CSS-цепочки — как последний fallback. Учитывать динамические id (напр. React `useId` вида `#:r7g:`) — экранировать и не считать их стабильными между перезагрузками.

**Открытая техническая развилка (решить при реализации, M2):** как sub-agent «видит» страницу, не взрываясь по токенам. Сырой `page.content()` огромен. Варианты:
- (a) Playwright accessibility snapshot (`page.accessibility.snapshot()`) как компактная основа + сырой HTML фрагмент только для конкретного узла по запросу.
- (b) Предобработка DOM: вырезать `script/style/svg-path`, оставить видимые + интерактивные элементы с атрибутами (role, aria, text, id, data-*, class), схлопнуть.

Стартовать с (a) как более дешёвого и структурного; (b) добавлять точечно, если a11y-дерево не даёт нужного селектора. Это ключевое место для итерации — не выдавать первую версию за финал.

### 4.4. Роль скриншотов

Скриншоты — в первую очередь для **видимого лога** (чтобы на демо-видео была видна работа) и как опциональный vision-fallback на визуально неоднозначных страницах (canvas, картинки без a11y). Основной канал перцепции — текстовые ответы sub-agent. Это позволяет держать оркестратор на дешёвой текстовой модели и не платить за image-токены на каждом шаге. Vision включать только если текстовая перцепция стабильно не справляется.

### 4.5. Управление контекстом (жёсткое требование ТЗ)

- **Наблюдения — только выжимки**, не сырой контент страниц.
- **История:** хранить полный лог отдельно; в контекст LLM подавать последние N шагов детально + rolling summary более ранних. N и порог суммаризации — в конфиге (стартовая прикидка N≈8).
- **Prompt caching:** системный промпт + tool schemas держать стабильным префиксом, чтобы попадали в кэш (кэш-хит ≈10% от цены входа у обоих провайдеров).
- **Жёсткие лимиты на длину** любого текста из страницы перед вставкой в контекст (обрезка до X символов, X в конфиге).

### 4.6. Провайдер-абстракция

Тонкий интерфейс:
```ts
interface LLMProvider {
  complete(req: {
    system: string;
    messages: Message[];
    tools: ToolSchema[];
  }): Promise<{ toolCall?: ToolCall; text?: string; usage: Usage }>;
}
```
Реализации: `OpenAIProvider` (основная), `AnthropicProvider` (опционально). Модель/провайдер оркестратора и sub-agent задаются независимо в конфиге.

## 5. Обработка ошибок (второй продвинутый паттерн)

Агент должен **адаптироваться**, а не падать:
- Тул вернул ошибку (селектор не найден / элемент не кликается / навигация зафейлилась) → ошибка кладётся в контекст как результат, оркестратор переосмысляет: повторный `query_dom`, `scroll`, `wait` загрузки, иной подход.
- **Устаревший селектор** после навигации/перерисовки → пере-запросить у sub-agent, а не переиспользовать старый.
- **Детект зацикливания:** если K шагов подряд без прогресса (одинаковые действия/состояние) → сменить стратегию или `ask_user`. K в конфиге.
- Таймауты навигации и ожидания элементов.

---

## 6. Security-гейт (третий продвинутый паттерн)

Перед необратимым/деструктивным действием — **пауза и подтверждение у пользователя** в терминале (показать, что именно собирается сделать, ждать y/n).

**Как определять деструктивность БЕЗ хардкода конкретных кнопок** (нельзя хардкодить «кнопка Оплатить»): семантическая оценка намерения действия перед `click`/`type`/submit — лёгкий классификатор (может быть отдельным дешёвым LLM-вызовом или частью рассуждения оркестратора), который по контексту (текст/роль элемента + текущая цель) решает, похоже ли действие на необратимое: оплата, покупка, удаление, отправка сообщения/письма, подтверждение заказа. Это **оценка в рантайме, не список кнопок.**

Демо должно явно показать эту остановку (напр. «дошёл до корзины, оплату не произвожу — подтвердите»).

## 7. Persistent-сессии

- `launchPersistentContext(userDataDir, { channel: 'chrome', headless: false })`.
- Пользователь логинится в нужные сервисы **вручную один раз**; профиль (куки/сессии) сохраняется между запусками.
- Путь профиля — в конфиге; предусмотреть флаг для сброса профиля.

## 8. Защитные лимиты (стоп-условия)

Все — в конфиге, значения ниже — стартовые **прикидки**:
- `maxSteps` ≈ 40 — предел итераций на задачу.
- `maxConsecutiveErrors` ≈ 5.
- `maxNoProgress` (K) ≈ 4.
- `stepTimeoutMs`, `navTimeoutMs`.

## 11. Демо-видео (результат сдачи)

**Задача для записи** (стабильнее, чем hh.ru с модерацией/капчей): заказ еды на Яндекс.Лавке/Еде. Пример промпта: «Открой Яндекс.Лавку, выбери мой рабочий адрес (он сохранён), найди какой-нибудь хот-дог, который сюда доставляют, и добавь его в корзину. Оплату не производи.»

**На видео одновременно видно:** терминал (ввод задачи + поток вызовов тулов с аргументами + работа sub-agent) и браузер (переходы, клики, ввод). Финал — сводка агента и явная остановка перед оплатой. Приложить ссылку на репозиторий.

## 12. Чего НЕ должно быть (жёсткие анти-требования)

- ❌ Захардкоженные селекторы (`a[data-qa='vacancy']` и подобное). Селектор всегда добывается в рантайме.
- ❌ Заготовленные последовательности действий/планы под конкретные задачи (шаги «оформить заказ», «удалить спам»). Агент строит план в моменте.
- ❌ Подсказки по ссылкам/текстам: не хардкодить, что вакансии на `/vacancies`, что добавить в корзину = кнопка «Заказать». Агент выясняет это сам.
- ❌ Отправка целых веб-страниц в контекст модели.

## 13. Порядок реализации (milestones для исполнителя)

- **M0 — Скелет.** Проект на TS, Playwright поднимает видимый persistent-браузер, терминал-REPL принимает строку. Коммит.
- **M1 — Луп + провайдер.** Абстракция `LLMProvider` (OpenAI), один тул `navigate`, минимальный оркестратор-луп. Проверка: модель вызывает тул по NL-задаче.
- **M2 — Перцепция + sub-agent.** `perception.ts` + `dom-agent.ts`; тулы `query_dom`/`click`/`type`. Агент кликает по селекторам, найденным в рантайме. Решить развилку 4.3.
- **M3 — Контекст.** Обрезка истории + rolling summary + prompt caching; `scroll`/`wait`/`read_page`.
- **M4 — Ошибки.** Retry/переосмысление, устаревшие селекторы, детект зацикливания, таймауты.
- **M5 — Security + финал.** Гейт необратимых действий, `ask_user`, `done` с отчётом.
- **M6 — Демо.** Прогон задачи заказа еды, стабилизация (несколько прогонов), README, запись видео.

## 14. Заметки для исполнителя

- Работай инкрементально по milestones, не пытайся собрать всё сразу. После каждого — рабочий коммит.
- При неопределённости в API Playwright/провайдеров — сверяйся с актуальной документацией, не по памяти.
- Sub-agent-перцепция (4.3) — самое рискованное место по токенам и надёжности; заложи время на итерацию, первую версию не считай финальной.
- Держи логи в терминале читаемыми (формат `Using tool / Input / Result`, ответы sub-agent, рассуждения) — это часть демонстрации.
